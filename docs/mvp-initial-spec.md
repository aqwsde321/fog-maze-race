# Fog Maze Race 초기 MVP 명세서

프로젝트 시작 시점에 사용한 초안 명세서입니다. 현재 구현과 차이가 있을 수 있으므로, 역사적 기준 문서로 보관합니다.

## 1. 개요

여러 사용자가 동일한 미로에서 동시에 출발하여 제한된 시야(Fog of War) 상태로 골인 지점까지 이동하는 실시간 멀티 레이스 게임

## 2. 전체 흐름

```text
[닉네임 입력]
→ [방 목록]
→ [방 생성 / 입장]
→ [게임 화면 (waiting)]
→ [카운트다운]
→ [게임 진행 (playing)]
→ [결과 화면 (6초)]
→ [waiting 복귀]
```

## 3. 사용자 정책

### 닉네임

- 필수 입력
- 길이: 1~5자
- 중복 허용
- localStorage 저장

### 플레이어 식별

- `playerId` (UUID)
- `nickname`
- `color` (방 입장 시 할당)

## 4. 방 시스템

### 방 생성

- 사용자 생성
- 생성자가 방장

### 방 입장

- `roomId` 기반
- `waiting` 상태일 때만 입장 가능
- 최대 인원: 15명

### 방 목록 표시

- 방 이름
- 방장 이름
- 현재 인원
- 상태 (`waiting` / `countdown` / `playing` / `ended`)

### 방 상태

- `waiting`
- `countdown`
- `playing`
- `ended`

## 5. 게임 화면 구조

### 단일 화면 구조

- 대기실 없음
- 방 입장 시 바로 게임 화면 진입

### 구성

- 상단: 상태 / 카운트다운
- 중앙: 게임 맵(Canvas)
- 우측: 플레이어 목록
- 버튼:
  - 시작 (방장만)
  - 나가기

## 6. 게임 시작

### 시작 조건

- 방장만 시작 가능
- 최소 인원 제한 없음

### 시작 흐름

```text
START_GAME
→ status = countdown
→ COUNTDOWN(3,2,1,0)
→ status = playing
```

### 카운트다운

- 서버가 매초 이벤트 전송

## 7. 맵 구조

### 구성

```text
[Start Zone] → [Maze] → [Goal Zone]
```

### 특징

- 시작 영역: 미로 외부
- 단일 입구로 미로 연결
- 도착 영역: 항상 공개

### 맵 방식

- 사전 제작 맵
- 여러 개 중 랜덤 선택

## 8. 플레이어 위치

### 시작

- 시작 영역 고정 슬롯 배치

### 이동

- 상하좌우 1칸 이동
- 벽 통과 불가
- 플레이어 충돌 없음

## 9. 시야 시스템

### 기본

- 미로 영역만 Fog 적용
- 시야 크기: 7x7

### 표시 규칙

#### 시작 영역 / 도착 영역

- 항상 공개
- 플레이어도 항상 표시

#### 미로 영역

- 시야 안에 있는 플레이어만 표시

### 골인 플레이어

- 전체 맵 공개
- 모든 플레이어 위치 표시
- 클라이언트 렌더링 모드 변경으로 처리

### 구현 방식

- 서버: 모든 플레이어 위치 전송
- 클라이언트: visibility 판단 후 렌더링

## 10. 이동 처리

### 방식

- 서버 authoritative

### 흐름

```text
클라이언트 → MOVE(direction)
서버 → 검증 → 좌표 업데이트
서버 → PLAYER_MOVED broadcast
```

### 전송 데이터

- 방향만 전송
- 좌표는 서버가 계산

## 11. 게임 종료

### 조건

- 현재 방에 남아있는 플레이어 전원 골인

### 예외

- 나간 플레이어는 제외

### 결과 화면

- 표시 시간: 6초
- 순위 표시
- 완주 / 나감 표시
- 이후 `waiting` 상태로 초기화

## 12. 플레이어 상태

- `waiting`
- `playing`
- `finished`
- `disconnected`
- `left`

## 13. 연결/이탈 처리

### 나가기 버튼

- 즉시 제거
- 결과에 `나감` 표시
- 재접속 불가

### 연결 끊김

- `disconnected` 상태
- 30초 유예

#### 30초 내 복귀

- 상태 복구
- 위치 유지

#### 30초 초과

- `left` 처리

## 14. 방장 정책

### 방장 권한

- 시작
- 강제 종료
- 방 이름 수정

### 방장 이탈

- 자동 위임 (입장 순 기준)

## 15. 통신 방식

- WebSocket 사용

## 16. 이벤트 구조

### Client → Server

- `CONNECT`
- `CREATE_ROOM`
- `JOIN_ROOM`
- `LEAVE_ROOM`
- `START_GAME`
- `MOVE`

### Server → Client

- `CONNECTED`
- `ROOM_LIST_UPDATE`
- `ROOM_JOINED`
- `ROOM_LEFT`
- `ROOM_STATE_UPDATE`
- `GAME_STARTING`
- `COUNTDOWN`
- `PLAYER_MOVED`
- `PLAYER_FINISHED`
- `GAME_ENDED`
- `ERROR`

## 17. 렌더링

### 플레이어 표시

- 기본: 원형
- 본인: 테두리 강조

### 확장 가능

- 스프라이트 교체 가능 구조 유지

## 18. MVP 범위

### 포함

- 멀티플레이
- 이동
- 시야 제한
- 골인 및 순위
- 방 시스템
- WebSocket 실시간 동기화

### 제외

- 애니메이션
- 채팅
- 아이템
- 커스터마이징
- 리플레이

## 19. 핵심 설계 원칙

- 서버 authoritative
- 클라이언트는 렌더링 중심
- 상태와 표현 분리
- 이벤트는 최소화 + 스냅샷 보조

## 20. 목표

- 빠른 플레이 (1~2분)
- 직관적 규칙
- 멀티플레이 경쟁 경험
- MVP 빠른 구현
