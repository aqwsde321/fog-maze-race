# 기능 명세: Fog Maze Race MVP

**기능 브랜치**: `001-fog-maze-race`
**작성일**: 2026-03-22
**상태**: 초안
**입력**: 사용자 설명: "여러 사용자가 동일한 미로에서 제한된 시야로 경쟁하는 실시간 멀티 레이스 게임 MVP"

## 사용자 시나리오와 테스트 *(필수)*

### 사용자 스토리 1 - 방에 들어가 레이스를 완주한다 (우선순위: P1)

플레이어는 닉네임을 입력한 뒤 방을 만들거나 대기 중인 방에 입장하고, 단일 게임 화면에서
카운트다운 후 미로를 이동해 골인 지점까지 도착할 수 있어야 한다. 경기 종료 후에는 순위와
이탈 여부를 확인하고 같은 방에서 다음 판 대기 상태로 돌아갈 수 있어야 한다.

**이 우선순위인 이유**: 이 흐름이 제품의 핵심 가치인 "짧고 직관적인 실시간 경쟁 경험"을 직접
제공하는 최소 플레이 가능 범위이기 때문이다.

**독립 검증 방법**: 두 명 이상의 사용자가 닉네임 입력 후 같은 방에 입장해 경기 시작,
이동, 골인, 결과 확인, 대기 상태 복귀까지 완료하면 독립적으로 검증할 수 있다.

**인수 시나리오**:

1. **조건** 플레이어가 닉네임을 입력하고 대기 중인 방에 입장해 있다, **행동** 방장이
   경기를 시작한다, **결과** 모든 참가자는 동일한 카운트다운을 보고 게임 진행 상태로
   전환된다.
2. **조건** 경기가 진행 중이다, **행동** 플레이어가 상하좌우로 이동해 골인 지점에
   도달한다, **결과** 해당 플레이어의 완주 순위가 기록되고 완주자는 전체 맵과 모든
   플레이어 위치를 볼 수 있다.
3. **조건** 현재 방에 남아 있는 모든 플레이어의 경기가 끝났다, **행동** 결과 화면
   표시 시간이 종료된다, **결과** 방은 다음 판을 준비하는 대기 상태로 초기화된다.

---

### 사용자 스토리 2 - 연결이 끊겨도 경기 상태를 복구한다 (우선순위: P2)

플레이어는 경기 중 네트워크 연결이 잠시 끊기더라도 제한 시간 안에 복귀하면 기존 방,
기존 위치, 기존 진행 상태를 이어서 플레이할 수 있어야 한다. 복귀하지 못한 플레이어는
경기 종료 계산에서 제외되어야 한다.

**이 우선순위인 이유**: 실시간 멀티플레이에서 일시적인 연결 끊김은 흔하며, 이를 복구하지
못하면 경기 공정성과 사용자 경험이 크게 훼손된다.

**독립 검증 방법**: 경기 진행 중 한 플레이어의 연결을 끊은 뒤 30초 이내 재접속과
30초 초과 미복귀를 각각 시험해 상태 복구와 이탈 처리를 확인하면 된다.

**인수 시나리오**:

1. **조건** 플레이어가 경기 중 연결이 끊겼다, **행동** 30초 이내에 동일한 사용자로
   복귀한다, **결과** 시스템은 해당 플레이어를 같은 방과 같은 경기 상태로 복구하고
   마지막 위치를 유지한다.
2. **조건** 플레이어가 연결이 끊긴 뒤 30초 안에 복귀하지 못했다, **행동** 유예 시간이
   끝난다, **결과** 시스템은 해당 플레이어를 경기에서 나간 것으로 처리하고 경기 종료
   조건 계산에서 제외한다.
3. **조건** 플레이어가 나가기 버튼으로 방을 떠난다, **행동** 방 이탈이 처리된다,
   **결과** 해당 플레이어는 즉시 제거되고 결과에는 "나감"으로 표시되며 같은 경기에는
   다시 들어올 수 없다.

---

### 사용자 스토리 3 - 방을 관리하고 다음 판을 이어서 진행한다 (우선순위: P3)

플레이어는 방 목록에서 현재 참가 가능한 방을 확인할 수 있어야 하며, 방장은 방 이름을
관리하고 경기를 시작하거나 강제로 종료할 수 있어야 한다. 방장이 떠나면 남은 참가자 중
가장 먼저 입장한 플레이어에게 방장 권한이 넘어가야 한다.

**이 우선순위인 이유**: 반복 플레이와 방 운영 편의성은 MVP 이후의 재사용성과 운영 안정성을
높이지만, 핵심 레이스 경험 자체보다는 우선순위가 낮다.

**독립 검증 방법**: 여러 방을 생성한 뒤 방 목록 표시, 방 이름 수정, 방장 위임,
강제 종료 후 대기 상태 복귀를 확인하면 독립적으로 검증할 수 있다.

**인수 시나리오**:

1. **조건** 사용자가 방 목록을 보고 있다, **행동** 여러 방이 존재한다, **결과** 각 방은
   방 이름, 방장 이름, 현재 인원, 현재 상태를 함께 표시한다.
2. **조건** 현재 방장이 방을 떠난다, **행동** 방 안에 다른 플레이어가 남아 있다,
   **결과** 가장 먼저 입장한 남은 플레이어가 새 방장이 된다.
3. **조건** 방장이 진행 중인 경기를 강제로 종료한다, **행동** 종료 처리가 완료된다,
   **결과** 방은 결과를 보여준 뒤 다시 대기 상태로 돌아간다.

### 엣지 케이스

- 정원이 15명인 방에는 추가 입장이 거절되어야 한다.
- 경기 상태가 `countdown`, `playing`, `ended` 인 방에는 새로 입장할 수 없어야 한다.
- 두 명 이상의 플레이어가 거의 동시에 골인하더라도 결과는 서버가 인정한 도착 순서대로
  일관되게 결정되어야 한다.
- 방장이 카운트다운 또는 경기 중 떠나더라도 방장 권한은 자동 위임되고 경기는 계속되어야
  한다.
- 마지막으로 남은 미완주 플레이어가 연결 끊김으로 유예 시간을 초과하면 경기는 종료될 수
  있어야 한다.
- 완주한 플레이어는 전체 맵을 보더라도 아직 미완주인 다른 플레이어의 진행 상태를 바꾸지
  않아야 한다.

## 요구사항 *(필수)*

### 기능 요구사항

- **FR-001**: 시스템은 플레이어가 길이 1자 이상 5자 이하의 닉네임을 입력하도록 해야 한다.
- **FR-002**: 시스템은 닉네임 중복을 허용해야 하며, 같은 기기 또는 브라우저에서 다시
  방문했을 때 마지막으로 사용한 닉네임을 다시 불러와야 한다.
- **FR-003**: 시스템은 각 플레이어를 고유 식별자, 닉네임, 방 입장 시 부여되는 색상으로
  식별해야 한다.
- **FR-004**: 시스템은 사용자가 새 방을 만들 수 있게 해야 하며, 방을 만든 사용자를 방장으로
  지정해야 한다.
- **FR-005**: 시스템은 사용자가 대기 상태인 방에만 입장할 수 있게 해야 하며, 한 방의 최대
  수용 인원은 15명이어야 한다.
- **FR-006**: 시스템은 방 목록에서 각 방의 이름, 방장 이름, 현재 인원, 현재 상태를
  표시해야 한다.
- **FR-007**: 시스템은 방 상태를 `waiting`, `countdown`, `playing`, `ended` 네 가지로
  관리해야 한다.
- **FR-008**: 시스템은 플레이어가 방에 입장하면 별도 대기실 없이 단일 게임 화면으로
  진입시켜야 한다.
- **FR-009**: 시스템은 게임 화면에 현재 상태 또는 카운트다운, 미로 화면, 플레이어 목록,
  나가기 기능을 제공해야 하며, 시작 기능은 방장에게만 노출해야 한다.
- **FR-010**: 시스템은 방장만 경기를 시작할 수 있게 해야 하며, 경기 시작을 위한 최소 인원
  제한은 두지 않아야 한다.
- **FR-011**: 시스템은 경기 시작 시 `waiting` 상태를 `countdown` 으로 바꾸고, 3초
  카운트다운 후 `playing` 상태로 전환해야 한다.
- **FR-012**: 시스템은 각 경기마다 사전에 준비된 미로 중 하나를 무작위로 선택해야 한다.
- **FR-013**: 시스템은 맵을 분리된 시작 영역, 시작 영역과 미로를 잇는 세로 연결 통로,
  `25x25` 고정 미로 영역, 미로 내부의 단일 골 타일로 구성해야 한다.
- **FR-014**: 시스템은 플레이어를 시작 영역의 고정 슬롯에 배치해야 한다.
- **FR-015**: 시스템은 플레이어가 상하좌우로 한 칸씩 이동하도록 해야 하며, 벽 통과를
  허용하지 않아야 한다.
- **FR-016**: 시스템은 플레이어 간 위치 충돌로 이동이 막히지 않게 해야 한다.
- **FR-017**: 시스템은 미로 영역에만 시야 제한을 적용해야 하며, 기본 시야 범위는 플레이어
  중심 7x7 이어야 한다.
- **FR-018**: 시스템은 `waiting` 상태에서는 시작 영역과 시작 슬롯에 선 플레이어만
  프리뷰로 보여줘야 한다.
- **FR-019**: 시스템은 경기 중 시작 영역, 연결 통로, 골 타일을 계속 구분할 수 있게
  표시해야 하며, 미로 영역의 플레이어는 현재 시야 안에 있을 때만 보이게 해야 한다.
- **FR-020**: 시스템은 완주한 플레이어에게 전체 맵과 모든 플레이어 위치를 공개해야 한다.
- **FR-021**: 시스템은 완주 순위를 기록해야 하며, 현재 방에 남아 있는 플레이어 전원이
  완주했을 때 경기를 종료해야 한다.
- **FR-022**: 시스템은 방을 떠난 플레이어를 경기 종료 계산에서 제외해야 하며, 결과 화면에
  완주 여부 또는 나감 여부를 함께 표시해야 한다.
- **FR-023**: 시스템은 경기 종료 후 결과 화면을 6초 동안 보여준 뒤 방을 다시 `waiting`
  상태로 초기화해야 한다.
- **FR-024**: 시스템은 플레이어 상태를 `waiting`, `playing`, `finished`,
  `disconnected`, `left` 로 관리해야 한다.
- **FR-025**: 시스템은 나가기 버튼을 누른 플레이어를 즉시 방에서 제거하고, 같은 진행 중
  경기에는 다시 합류할 수 없게 해야 한다.
- **FR-026**: 시스템은 연결이 끊긴 플레이어를 `disconnected` 상태로 두고 30초 동안
  복귀 기회를 제공해야 한다.
- **FR-027**: 시스템은 끊긴 플레이어가 30초 이내 복귀하면 기존 방, 위치, 경기 상태를
  복원해야 한다.
- **FR-028**: 시스템은 끊긴 플레이어가 30초 안에 복귀하지 못하면 `left` 상태로 바꾸고
  경기에서 제외해야 한다.
- **FR-029**: 시스템은 방장에게 경기 시작, 방 이름 수정, 강제 종료 권한을 제공해야 한다.
- **FR-030**: 시스템은 방장이 방을 떠나면 남은 플레이어 중 입장 순서가 가장 빠른 사람에게
  방장 권한을 자동 위임해야 한다.
- **FR-031**: 시스템은 방 목록과 방 내부 상태를 모든 관련 사용자에게 실시간으로 동기화해야
  한다.

### 핵심 엔터티 *(데이터가 포함되는 기능이면 필수)*

- **Player**: 개별 참가자. 고유 식별자, 닉네임, 색상, 현재 방 소속, 현재 상태, 현재 위치,
  입장 순서, 연결 유예 만료 시각을 가진다.
- **Room**: 플레이어가 모이는 단위 공간. 방 이름, 방장, 현재 상태, 최대 인원, 참가자 목록,
  현재 경기 정보, 최근 결과를 가진다.
- **Match**: 한 번의 레이스 세션. 선택된 맵, 카운트다운 상태, 시작 시각, 완주 순서,
  진행 중 플레이어 목록, 종료 시각을 가진다.
- **Map Definition**: 사전 제작된 미로 정보. 시작 영역, 미로 영역, 도착 영역, 벽 배치,
  시작 슬롯, 세로 연결 통로, 미로 내부의 단일 골 타일을 포함한다.
- **Result Entry**: 경기 종료 후 표시에 쓰이는 항목. 플레이어 식별 정보, 최종 상태,
  완주 순위 또는 나감 여부를 포함한다.

## 가정

- 한 사용자는 동시에 하나의 방에만 참여한다.
- 관전자 모드는 MVP 범위에 포함하지 않는다.
- 한 방에서는 동시에 하나의 경기만 진행한다.
- 방장의 강제 종료는 현재 경기를 즉시 끝내고, 현재까지 확정된 결과와 나감 상태를 바탕으로
  결과 화면을 보여준 뒤 대기 상태로 복귀하는 것으로 간주한다.
- 방 이름 길이와 문자 제한은 MVP에서 별도 제약을 두지 않고 일반적인 텍스트 입력으로 다룬다.

## 아키텍처 제약 *(필수)*

- **AC-001**: **권한 모델**: 경기 규칙 판정, 플레이어 위치, 방 상태 전환, 완주 순위는
  서버가 최종 기준이어야 한다.
- **AC-002**: **클라이언트 역할**: 클라이언트는 닉네임 입력, 방 선택, 이동 입력, 상태 표시,
  시야 규칙에 따른 렌더링에 집중해야 하며 경기 결과를 독자적으로 확정하면 안 된다.
- **AC-003**: **상태 경계**: 방 상태, 경기 상태, 플레이어 상태 같은 도메인 상태는
  표현 상태와 분리되어 관리되어야 하며, 화면 표시 방식이 경기 규칙을 바꾸면 안 된다.
- **AC-004**: **실시간 복구**: 실시간 세션은 상태 전환과 이동을 점진적으로 반영할 수
  있어야 하며, 연결 복구 시에는 최신 방 또는 경기 스냅샷으로 동일한 상태를 재구성할 수
  있어야 한다.
- **AC-005**: **MVP 제외 범위**: 애니메이션, 채팅, 아이템, 커스터마이징, 리플레이는 첫
  출시 범위에서 제외해야 한다.

## 범위 제외

- 캐릭터 스프라이트 교체를 제외한 커스터마이징 기능
- 경기 중 채팅 또는 감정 표현 시스템
- 아이템, 함정, 버프 같은 추가 규칙
- 경기 리플레이, 관전, 기록 분석 기능
- 고급 연출용 애니메이션과 시네마틱 효과

## 성공 기준 *(필수)*

### 측정 가능한 결과

- **SC-001**: 처음 접속한 사용자가 닉네임 입력 후 방 생성 또는 입장을 완료하고 경기 시작을
  기다리는 화면까지 30초 안에 도달할 수 있다.
- **SC-002**: 한 방에서 최대 15명의 플레이어가 동일한 경기 상태 전환을 경험하며 한 판을
  문제 없이 시작하고 종료할 수 있다.
- **SC-003**: 완주가 발생한 모든 경기에서 결과 화면은 100%의 완주 플레이어에게 명확한
  순위를, 이탈 플레이어에게 명확한 상태 표시를 제공한다.
- **SC-004**: 경기 중 연결이 끊긴 플레이어가 30초 안에 복귀한 경우, 별도의 관리자 개입
  없이 같은 방과 같은 진행 상태로 복구된다.
- **SC-005**: 제공된 미로 기준으로 일반적인 경기 한 판은 결과 화면과 대기 상태 복귀까지
  포함해 2분 안에 마무리될 수 있다.
- **SC-006**: MVP 데모에서 참가자는 별도 설명 없이 "닉네임 입력 → 방 입장 → 이동 →
  골인 → 결과 확인" 흐름을 완수할 수 있어야 한다.
